# Roadmap Construction

## Purpose
Group work into milestones that deliver visible progress while reducing delivery risk.

## Milestone Rules
- Each milestone should produce a coherent, demonstrable increment.
- Early milestones should reduce uncertainty or unlock parallel work.
- Later milestones should build on stable interfaces and validated groundwork.
- A milestone should not depend on undefined work outside the roadmap.

## How To Build A Roadmap
1. Identify the end-state and the smallest meaningful intermediate outcomes.
2. Cluster tasks by dependency and user-visible value.
3. Put risk-reducing or architecture-setting work early.
4. Separate foundation, feature delivery, integration, and hardening when helpful.
5. Check that each milestone can be explained in one sentence.

## Common Milestone Shapes
- Foundation: structure, contracts, tooling, or baseline docs
- Core capability: first working slice of the main feature
- Integration: cross-system wiring, migrations, or external dependencies
- Hardening: quality, reliability, performance, or operational readiness

## Warning Signs
- A milestone contains unrelated work only because it happens at the same time.
- Early milestones produce no usable or reviewable output.
- Dependencies flow backward from later milestones to earlier ones.
- The roadmap measures activity instead of delivery.

## Output Pattern
- `Milestone`: short name
- `Goal`: what becomes possible after completion
- `Included work`: the tasks or themes grouped there
- `Exit criteria`: how completion is recognized
